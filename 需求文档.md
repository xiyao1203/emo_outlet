# 情绪出口（Emo Outlet）产品需求文档

> 一个"可以安全骂人，但不会伤害任何人的 AI 出气筒"

---

## 1. 产品概述

### 1.1 产品名称

**情绪出口（Emo Outlet）**（暂定）

### 1.2 产品定位

一款基于 AI 的安全情绪释放工具，通过虚拟对象、对话互动、方言表达和情绪可视化，帮助用户释放压力、整理情绪。

### 1.3 核心价值

- **安全宣泄** — 替代现实冲突
- **情绪表达** — 文字 + 语音 + 方言
- **AI 承接** — 不评判、不升级冲突
- **可视化输出** — 海报

### 1.4 使用场景

- 被老板骂但不能反击
- 和伴侣吵架后情绪积压
- 客户/同事带来的压力
- 日常情绪发泄需求

### 1.5 项目可行性总评

```
✅ 产品方向可行     — 情绪宣泄 + AI 对话 + 方言，市场空白明确
✅ 技术方案可行     — 主流 AI API + Flutter 成熟技术栈
⚠️ 最大风险在合规   — 需谨慎处理敏感话题，做好合规包装
⚠️ 方言 TTS 是长期战 — 一期用文本方案过渡
✅ 差异化足够强     — 竞品不覆盖的核心场景明确
✅ MVP 路径清晰     — 按迭代计划可落地
```

## 2. 竞品分析

### 2.1 竞品选择

#### 1️⃣ Replika

AI 虚拟陪伴 + 情绪对话，强人格化，有长期关系养成。

- **优点**：AI 对话自然、情绪承接能力强、用户粘性高
- **缺点**：不支持"情绪对抗"、无宣泄场景、无方言能力

#### 2️⃣ Character.AI

多角色 AI 对话，用户自定义角色。

- **优点**：角色丰富、可玩性强、对话多样
- **缺点**：情绪安全弱、无专门宣泄设计、无时间控制机制

#### 3️⃣ Wysa

心理健康引导，CBT（认知行为疗法）。

- **优点**：强心理安全、情绪引导专业
- **缺点**：偏理性不适合发泄、没有"骂/吐槽"场景

#### 4️⃣ Soul

情绪社交平台。

- **优点**：用户基数大、情绪表达丰富
- **缺点**：不私密、有社交压力、无 AI 承接

### 2.2 竞品总结

| 维度 | 现有产品 | 本产品机会 |
|------|---------|-----------|
| 情绪发泄 | ❌ 几乎没有 | ✅ 核心功能 |
| 方言支持 | ❌ | ✅ 差异化 |
| AI 对抗 | ❌ | ✅ |
| 时间控制 | ❌ | ✅ |
| 私密性 | ⚠️ 部分 | ✅ |
| 海报输出 | ❌ | ✅ |

> **核心差异：我们是"可控的、安全的情绪对抗系统"，不是聊天工具**

### 2.3 竞品分析补充

| 竞品 | 与本产品关系 | 值得借鉴 |
|------|-------------|---------|
| **Pi**（by Inflection AI） | AI 情绪对话 | 极其自然的倾听式对话风格，可作为"单向模式"的参考标杆 |
| **Hugging Face Chat** | 可自定义角色的 AI 对话 | 角色设定 Prompt 工程思路 |
| **小睡眠 / 潮汐** | 情绪疏导工具 | 治愈系 UI 设计、引导流程 |
| **微软小冰** | 中文情绪对话 | 中文情感理解能力强，方言/语音交互成熟 |

## 3. 产品功能设计

### 3.1 用户系统

- 手机号登录
- 邮箱登录
- 游客模式
- 注销账号

### 3.2 泄愤对象系统

**创建字段：**

```json
{
  "name": "讨厌的老板",
  "type": "boss",
  "appearance": "中年男性，西装",
  "personality": "爱甩锅",
  "relationship": "直属领导",
  "style": "漫画"
}
```

**输出：** AI 生成虚拟形象（非真人）

### 3.3 泄愤模式

**单向模式：** 用户输出情绪，AI 只承接

**双向模式：** AI 可反驳

**风格可选：**

| 风格 | 描述 |
|------|------|
| 嘴硬型 | 不认错 |
| 道歉型 | 软化冲突 |
| 冷漠型 | 不在乎 |
| 阴阳型 | 带点嘲讽 |
| 理性型 | 讲道理 |

### 3.4 方言系统

**一期支持：**

- 普通话
- 粤语
- 四川话
- 东北话
- 上海话

**技术策略：**

1. 文本风格转换（优先）
2. 语音识别（后续）
3. TTS 方言合成（长期）

### 3.5 输入方式

**文字：** 实时输入、长文本支持

**语音：** 语音转文字、方言识别、AI 语音回复（可选）

### 3.6 时间控制

**选项：** 1 分钟 / 3 分钟 / 5 分钟 / 10 分钟

**结束后：** 自动停止 → 生成总结

### 3.7 情绪分析

**输出：**

- 情绪类型
- 情绪强度
- 高频词
- 建议

### 3.8 海报生成

**内容：** 虚拟对象、情绪关键词、时间、插画

> ❌ 不展示骂人原文

### 3.9 分享系统

- 保存图片
- 微信分享
- 系统分享（Mac）

### 3.10 功能优化建议

#### 建议 1：增加"情绪速写"入口（快速出气模式）

完整流程路径较长，用户有情绪时可能没耐心走完。

- 点击即进入默认匿名对话
- 默认倒计时 3 分钟
- 结束后生成简易版海报
- 引导用户后续创建专属对象

#### 建议 2：泄愤对象增加"AI 补全"

- 用户只需输入目标名称和关系（如"张经理"+"直属领导"）
- AI 自动补全外貌、性格、对话风格
- 用户可微调而非从零创建

#### 建议 3：多样化分享内容

- **情绪便签**：短文案 + 小插画，适合朋友圈
- **周报模式**：一周情绪趋势图
- **匿名故事卡片**：对话提炼为脱敏故事

#### 建议 4：情绪温度计

对话界面实时显示情绪强度指示器（温度计/表情变化），直观感受情绪变化曲线。

## 5. 技术架构

```
客户端（Flutter）
  ├── Android
  └── macOS

后端
  ├── 用户服务
  ├── 会话服务
  ├── AI 编排
  └── 海报服务

AI 能力
  ├── LLM 对话
  ├── 图像生成
  ├── ASR
  └── TTS

存储
  ├── MySQL
  ├── Redis
  └── OSS

### 5.1 技术可行性评估

| 模块 | 可行性 | 说明 |
|------|--------|------|
| LLM 对话引擎 | ✅ **高** | 国内外大模型 API（OpenAI / 通义千问 / DeepSeek）均支持 Prompt 工程实现单向/双向对话 |
| 方言文本转换 | ✅ **中高** | Prompt 风格控制 + 方言词库替换可实现，需投入方言语料整理 |
| 图像生成（虚拟形象） | ✅ **高** | Stable Diffusion / DALL·E / Midjourney API 均可实现 |
| 海报合成 | ✅ **高** | Canvas 类库（Python Pillow / Flutter CustomPainter）成熟 |
| 语音转文字（ASR） | ✅ **高** | 讯飞 / 阿里云 / Whisper 均支持中文及方言识别 |
| 方言 TTS | ⚠️ **中低** | 粤语 TTS 较成熟，四川话/东北话/上海话的 TTS 方案极少，需自研或等待行业成熟 |
| 实时对话 + 倒计时 | ✅ **高** | WebSocket + 后端计时器，技术方案成熟 |
| 情绪分析 | ✅ **中高** | LLM 可直接输出情绪分析结果，无需单独训练模型 |

> **最大技术瓶颈：方言 TTS（一期用文本方案过渡，语音方案留待长期）**

### 5.2 平台策略

**MVP 阶段只做一个平台，推荐优先顺序：**

1. **Android** — 国内安卓市场审核相对宽松，上架快，用户基数大
2. **macOS** — Flutter 开发成本低，Apple 开发者账号统一管理
3. **iOS（第二阶段）** — App Store 审核最严格，需做好合规包装再上
```

## 9. 安全设计（非常关键）

### 9.1 风险点

- 人身攻击
- 现实报复引导
- 自伤/他伤
- 隐私泄露

### 9.2 防护策略

- 敏感词过滤
- LLM 输出审核
- 高风险中断机制

### 9.3 高风险处理

检测到攻击/暴力意图：
→ 中断对话
→ 提供冷静建议
→ 引导用户离开

### 9.4 核心风险分析

#### AI 安全红线

LLM 在涉及暴力、色情、自伤等话题时可能触发厂商内容审核，导致 API 调用失败甚至封号。

**应对策略：**
- 主力选择审核较宽松的 LLM 供应商（如 DeepSeek、本地部署模型），OpenAI 等严格厂商作为兜底
- 建立本地敏感词库做第一道过滤
- 中长期本地部署 7B-13B 小模型（如 Qwen2.5-7B），LoRA 微调适配"情绪对抗"场景

#### 应用商店合规

App Store 和国内安卓商店对"发泄"类 app 审核严格。

**应对策略：**
- 产品对外包装为 **"情绪管理""心理健康""正念疏导"** 工具
- UI 风格走柔和、治愈路线
- 对话内容 **不上云明文存储**（端侧加密或即用即毁）

#### 用户沉迷 / 滥用风险

用户可能依赖 AI 出气筒，强化攻击性思维。

**应对策略：**
- 情绪报告加入 **正向引导**（如"今天释放了愤怒，也试试放松一下"）
- 免费版每日使用次数上限（3 次/天），兼顾商业化和防沉迷
- 连续使用 7 天后弹出情绪健康提示

### 9.5 数据与隐私策略

| 维度 | 建议方案 |
|------|---------|
| 对话内容 | **端侧加密存储**，服务器不保存明文；或对话结束后即销毁 |
| 用户标识 | 游客模式使用设备 UUID + 匿名 ID，不强制绑定手机号 |
| 数据分析 | 只上报脱敏后的情绪类型和强度统计，不上报原始对话 |
| 合规备案 | 如涉及"心理健康"类目，需关注《生成式AI服务管理办法》等相关法规 |

## 10. MVP 规划

### 第一阶段（8周迭代）

原计划 4 周风险过高，建议调整为 8 周迭代：

| 周次 | 核心交付 | 说明 |
|------|---------|------|
| **第 1 周** | 基础对话框架 | 用户系统 + 单向文字对话 + 倒计时，无形象无海报 |
| **第 2 周** | 泄愤对象 + Prompt 工程 | 创建对象 + 单向/双向模式切换 + 风格切换 |
| **第 3 周** | 形象生成 + 情绪分析 | AI 形象生成 + 会话结束后的情绪分析输出 |
| **第 4 周** | 海报生成 + 分享 + 打磨 | 海报合成 + 保存分享 + 整体体验优化 |
| **第 5-6 周** | 方言 + 语音 | 方言文本转换 + 语音输入 |
| **第 7-8 周** | 内测 + 修复 | 小范围内测，收集反馈修复问题 |

### 第二阶段

双向模式、语音、方言

### 第三阶段

AI 语音、情绪报告、会员系统
